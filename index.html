Phase 01 · 4–6 weeks
Discovery

Discovery maps the current landscape across all six tracks, then converges on an agreed target state and a concrete, backlog-ready plan. The phase runs in two modes: a divergent research sprint (weeks 1–3) where teams go wide — interviewing, auditing, and documenting — followed by a convergent design sprint (weeks 3–6) where findings are synthesised into a validated to-be design and prioritised roadmap.

Phase 03 · 8–16 weeks
Build

Build is the iterative delivery phase — 2-week sprints from Sprint 0 through to MVP and beyond. Sprint 0 validates the entire setup with one thin-slice end-to-end use case, proving that the platform, team, and process all work together before committing to the full delivery cadence. Subsequent sprints build, test, and demo against the prioritised backlog.

Foundation activities — what the team actually does
TechnologyOrg provisioning & platform setup
▾

Provision the three Salesforce environments and configure platform-level settings that everything else depends on. This is typically done by the technical lead in the first week of Foundation.

  • Request and provision Dev, QA, and UAT orgs with the correct Service Cloud edition (Unlimited, Enterprise, or Professional)
  • Configure My Domain, enable Lightning Experience, set session security settings
  • Install any required managed packages (CTI adapter, third-party AppExchange apps)
  • Set up Connected Apps and OAuth settings for integrations
  • Configure org-wide email address, from-name, and deliverability settings
  • Enable and configure Salesforce Knowledge, Chatter, and any other required features
TechnologyCore object model configuration
▾

Build the data model agreed in WS 05. This is the single most important Foundation activity — getting the object model right prevents expensive schema changes during Build.

  • Create Case record types (e.g. Technical Support, Billing, General Enquiry, Complaint) and configure page layouts per type
  • Add all custom fields to Case: category, sub-category, root cause, resolution code, contact channel, and any business-specific fields
  • Configure Account and Contact objects: custom fields, person accounts if required, household model if applicable
  • Create any custom objects identified in Discovery (e.g. Product, Asset, Entitlement Plan)
  • Configure Entitlements and Milestones for SLA tracking if in scope
  • Set up compact layouts, related lists, and highlights panels per console design spec
TechnologyOmni-Channel & routing foundation
▾

Set up the Omni-Channel framework so that work items can be routed and agents can receive work. Detailed routing logic (skills, capacity, fallback) is built in Sprint 1, but the foundation must exist first.

  • Enable Omni-Channel, configure the Service Console app with the Omni-Channel utility
  • Create service channels for all case origins: Email, Phone, Web, Chat, Social
  • Create initial queue structure matching the agreed routing model
  • Configure presence statuses and routing configurations
  • Set up Email-to-Case routing addresses and assign to queues
  • Set up Web-to-Case form and verify case creation and routing
TechnologyCTI / telephony integration
▾

Connect the telephony platform to Salesforce via Open CTI or a vendor-specific adapter. This is often the most complex integration and should be started in week 1 of Foundation.

  • Install and configure the CTI adapter (Amazon Connect, Genesys, Avaya, Nice, Five9, or vendor-specific)
  • Configure the softphone layout in the Service Console
  • Set up screen pop rules: match incoming ANI to Contact/Account, create case automatically on answer
  • Configure call logging: automatic activity creation on call end with duration, wrap-up code, and disposition
  • Test end-to-end: incoming call → screen pop → case creation → wrap-up → activity log
TechnologySecurity model implementation
▾

Implement the security model so agents, supervisors, admins, and any read-only stakeholders each see exactly what they need and nothing more.

  • Create profiles for each user type: Service Agent, Senior Agent, Supervisor, Admin, Read-Only
  • Create permission sets for additive permissions (e.g. "Can merge cases", "Can delete contacts")
  • Set up role hierarchy aligned to the org structure and supervisory model
  • Configure OWD settings for Case, Contact, Account — typically Private for Case, Public Read-Only for Account
  • Create sharing rules for any exceptions to OWD (e.g. team leads see all team cases)
  • Configure field-level security: restrict sensitive fields (e.g. payment reference, date of birth) to specific profiles
DeliveryBacklog refinement & team kickoff
▾

Prepare the team to start Sprint 0 immediately. All stories for the first two sprints should be refined, estimated, and have clear acceptance criteria before Foundation ends.

  • Break Discovery epics into user stories with business-facing acceptance criteria
  • Estimate stories using story points or hours — involve the full technical team
  • Identify and resolve any dependency or ambiguity blockers at story level
  • Set up the delivery tooling: Jira, Azure DevOps, or equivalent
  • Run team onboarding: walkthrough of the platform, the design decisions from Discovery, and the ways of working
  • Establish sprint ceremonies: planning, daily standup, review, and retro cadence and timing
Phase 03 · Sprint 0 + 6–8 sprints to MVP
Build & MVP
Build the product. Sprint by sprint, epic by epic. Demo real functionality every two weeks. Ship an MVP your agents can use.
Phase goal

Deliver a production-ready MVP that covers the core service operations: case creation across all channels, omni-channel routing, agent desktop, SLA tracking, basic automation, and a populated knowledge base. Sprint 0 validates the full stack. Subsequent sprints build the epics below in priority order.

Sprint 0 · 2 weeks · Validate before you build
Thin-slice end-to-end validation

Sprint 0 is not about building features. It proves the entire stack works together before the full team commits to the sprint cadence. Pick one contact reason (e.g. a billing query arriving via email) and build a complete end-to-end flow through the system — from case creation to closure. Everything else follows once this works.

Phase 01 · 3–4 weeks
Audit & Assessment
Before touching anything, understand what you have inherited. Map the org completely. Quantify the technical debt. Agree what to keep, fix, and replace.
Phase goal

Produce a complete, documented picture of the inherited Service Cloud org — what exists, what works, what is broken, and what is missing. This audit becomes the remediation backlog and the baseline for all future work. No assumptions, no shortcuts.

⚠️
The golden rule of account take-overs: Never make configuration changes to a live production org before the audit is complete and a full sandbox backup exists. Every change in a poorly documented org risks breaking something that was silently load-bearing.
Common risks inherited in a take-over
Undocumented automationFlows, triggers, and process builders that nobody knows the purpose of — but which are running in production.
Permissions sprawlProfiles and permission sets that have accumulated over years, granting users more access than they should have.
Data quality debtYears of inconsistent data entry: duplicates, missing required fields, and contacts with no associated account.
Tech debt in customisationApex triggers with no test coverage, hard-coded IDs in code, and deprecated API calls still running.
Unknown integrationsThird-party systems sending data to Salesforce via APIs that nobody on the current team set up or documented.
Missing documentationNo sandbox strategy, no deployment process, no change log. Every change is made directly to production.

Typical remediation activities
Priority 1Sandbox baseline & deployment process
▾

The single most important remediation activity. Without a sandbox and a deployment process, every change is a production risk.

  • Request a full sandbox refresh from production — this is your safety net
  • Establish the deployment process: change sets or SFDX from sandbox to production
  • Document the baseline state of the org before any changes are made
  • Agree a change freeze on production during the remediation phase
Priority 1Broken automation & integration fixes
▾

Automation that is erroring silently is causing data corruption or missed processes. Fix or deactivate before building anything new on top.

  • Review Flow and Apex error logs — fix all recurring errors
  • Deactivate Process Builder rules that are now redundant or causing conflicts
  • Fix integration failures identified in the audit — restore data sync if broken
  • Test critical automation end-to-end in sandbox before deploying fixes to production
Priority 2Data deduplication & quality
▾

Duplicate records undermine agent trust in the system. Deduplicate before go-live — not after.

  • Run duplicate jobs on Contacts and Accounts using agreed matching rules
  • Merge duplicates in batches — get business sign-off on merge strategy first
  • Enable duplicate rules to prevent new duplicates from being created
  • Bulk-update critical missing fields using Data Loader where possible
Priority 2Security & permissions cleanup
▾

Permission sprawl is a compliance risk. Tighten access now rather than after an incident.

  • Remove System Administrator profiles from non-admin users
  • Deactivate inactive users and reclaim unused licenses
  • Consolidate redundant profiles into a cleaner permission set model
  • Apply field-level security to sensitive data fields identified in the audit

Phase goal

⚠️
The golden rule of account take-overs: Never make configuration changes to a live production org before the audit is complete and a full sandbox backup exists. Every change in a poorly documented org risks breaking something that was silently load-bearing.
Common risks inherited in a take-over
Undocumented automationFlows, triggers, and process builders that nobody knows the purpose of — but which are running in production.
Permissions sprawlProfiles and permission sets that have accumulated over years, granting users more access than they should have.
Data quality debtYears of inconsistent data entry: duplicates, missing required fields, and contacts with no associated account.
Tech debt in customisationApex triggers with no test coverage, hard-coded IDs in code, and deprecated API calls still running.
Unknown integrationsThird-party systems sending data to Salesforce via APIs that nobody on the current team set up or documented.
Missing documentationNo sandbox strategy, no deployment process, no change log. Every change is made directly to production.
